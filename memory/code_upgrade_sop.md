## 1. 适用场景与核心原则

**适用**：GenericAgent 本地与官方上游（`lsdefine/GenericAgent`）同步、补齐功能、能力/记忆退役清理。

**核心原则（源自 INCOSE 系统配置管理 + 个人工程实践）**：
1. **资产优先**：本地定制（memory/skills/SOP/mykey/LLM编排模块）比官方代码珍贵。升级不可破坏资产。
2. **可回滚**：每次变更前必有可追溯的备份基线，支持 100% 回滚。
3. **可观测**：每步留痕（git log + tag + 报告），验证可复现。
4. **有进有出**：引入新能力的同时，识别并退役过时能力，防止腐化。

> 理论依据：INCOSE Systems Engineering Handbook（配置管理 + 基线管理）；DevOps GitOps（声明式 + 可回溯）。

## 3. 变更分类与评估门禁（ITIL 4 Change Enablement）

> 每次升级前，先对**变更**分类，决定执行路径。源自 ITIL 4 Change Enablement（标准/正常/紧急变更）+ ISO 31000 风险评估。

### 3.1 变更分类

| 类型 | 判定标准 | 执行路径 | 人工确认 |
|---|---|---|---|
| **标准变更**（Standard） | 官方改动的文件，**本地零定制**（git 自动合并无冲突）；非核心文件 | 可自动合并 | 否（记录即可） |
| **正常变更**（Normal） | 涉及核心框架（`agent_loop`/`llmcore`/`ga.py`/`agentmain`/`tools_schema`），或本地有定制 + 冲突 | dry-run 预览 → 人工逐个 diff → 合并 | **是** |
| **紧急变更**（Emergency） | 官方修复安全漏洞/严重 bug，需立即跟进 | 紧急流程（备份→合并→冒烟），跳过完整评估 | 事后补验证 |

### 3.2 评估门禁（Gate，合并前必过）

```
Step A: git fetch upstream
Step B: git merge-tree --write-tree --name-only HEAD upstream/main   # 只读冲突预览
Step C: 分类（标准/正常/紧急）+ 风险评级（低/中/高）
        - 风险因子：是否碰核心定制、改动行数、是否有 breaking change
Step D: 门禁判定
        - 全部"标准变更" + 风险"低" → 自动合并（先建备份tag）
        - 任一"正常变更" 或 风险"中/高" → 人工确认后再合并
        - 任何"紧急" → 走紧急流程
```

> 关键：**永不跳过 Step B**。dry-run 预览是避免破坏性 merge 的最后防线。

## 4. 备份与回滚策略（100% 可回滚）

> 源自 INCOSE 基线管理（配置基线 + 回退基线）。

### 4.1 备份分级（每次升级前至少 L1）

| 级别 | 方法 | 用途 | 生命周期 |
|---|---|---|---|
| **L1 整库基线** | `git tag backup/pre-upgrade-YYYYMMDD-HHMM` | 回退整次升级 | 见 §4.3 |
| **L2 单文件快照** | 备份到 `backups/<category>/`（带时间戳 .bak） | 单文件回滚（如快捷方式、配置） | 30天 |
| **L3 临时 stash** | `git stash push -u -m "..."` | 保护工作区未提交改动 | 合并验证后 pop+drop |

### 4.2 回滚分级

| 级别 | 场景 | 命令 | 副作用 |
|---|---|---|---|
| **R1 单文件** | 某文件合并后异常 | `git checkout <tag> -- <file>` | 仅该文件 |
| **R2 软回滚** | 整次升级有问题，保留新分支 | `git checkout -b recovery/YYYYMMDD <tag>` | 新分支隔离 |
| **R3 硬重置** | 灾难性，丢弃升级（最后手段） | `git reset --hard <tag>` | 丢弃升级后所有改动 |

> 策略：优先 R1（最小爆破半径），R3 仅在 R1/R2 无效时使用，且必须确认工作区已无未保存价值。

### 4.3 备份生命周期与清理（该备份就备份，无问题后清理）

```
清理门禁（全部满足才清理）：
1. 该备份对应的升级已运行 ≥ 30 天
2. 期间未触发任何回滚到该备份
3. 核心功能（TUI启动/code_run/agent循环）持续正常
4. 有更新的备份 tag 作为新基线
```

- 满足门禁后：删除旧 tag（`git tag -d <tag>`），保留最近 2 个备份 tag 作为滚动基线。
- 物理备份文件（`backups/*.bak`）：超 30 天且确认无用后删除。
- **禁止**：清理任何对应升级有未解决回滚的备份。

## 5. 标准升级流程

### 5.1 升级前
1. **建 L1 备份 tag**：`git tag backup/pre-upgrade-YYYYMMDD-HHMM`
2. **处理工作区**：`git stash -u` 或先 commit（工作区必须干净）
3. **枚举进程**：`psutil` 确认无活跃后台任务
4. **dry-run 门禁**（§3.2 Step A-D）

### 5.2 升级中
1. **合并**：`git merge upstream/main --no-ff -m "merge: sync upstream main <date>"`
2. **冲突处理原则**：
   - 纯官方文件（未定制）→ 接受 upstream
   - 核心框架（有定制）→ **必须人工三向 diff**，确认官方改动不破坏 LLM 编排/多 agent 定制
   - memory SOP → 保留本地实践经验 + 合并官方新增条目
   - TUI 文件 → 接受 upstream，但人工确认你曾 revert 过的功能是否需要
3. **冲突未解决期间**：文件带 `<<<<<<<` 标记，尽快解决或 `git merge --abort`，避免其他会话读到语法错误

## 6. 合并后验证体系（DevOps 金丝雀思想）

> 源自 DevOps 渐进式发布（冒烟→回归→灰度）。不中断服务下分层验证。

| 层级 | 验证内容 | 方法 | 通过标准 |
|---|---|---|---|
| **L0 静态检查** | 语法/冲突标记 | `py_compile` 全核心文件 + 搜 `<<<<<<<` | 零报错零残留 |
| **L1 单元导入** | 核心模块可加载 | `python -c "import agent_loop, llmcore"` | 无异常 |
| **L2 冒烟测试** | TUI 能启动 | `python frontends/tuiapp_v2.py` 短暂启动 | 界面正常显示 |
| **L3 回归测试** | 关键能力不退化 | 执行 1 次 `code_run` + 1 次 LLM 调用 | 结果符合预期 |
| **L4 灰度观察** | 实际使用 1-2 会话 | 用户正常使用半天 | 无异常报错 |

> 每层通过才进下一层（gate）。L0-L3 自动执行，L4 人工观察。验证结果记入升级报告。

## 8. 能力/信息生命周期管理（有进有出）

> 源自产品生命周期管理 + Gartner 技术成熟度曲线。**有引入就有退出**，防止记忆/能力腐化膨胀。

### 8.1 生命周期阶段

```
引入(Introduce) → 成熟(Mature) → 监控(Monitor) → 退役(Retire)
```

- **引入**：新增 SOP/记忆/能力时，**必须注明**：版本、创建时间、**预期复查日期**（建议 30-90 天）、适用场景。
- **成熟**：通过 ≥ 3 次真实任务使用验证，标记 `[verified]`。
- **监控**：定期评估 ROI（见下）。低 ROI 进入退役候选。
- **退役**：标记 `[DEPRECATED YYYY-MM-DD]`，**保留 14 天宽限期**，人工确认后删除。

### 8.2 过时能力识别（ROI 评估）

| 信号 | 判定 | 动作 |
|---|---|---|
| 长期未引用（>90天） | 可能过时 | 标记复查 |
| 与新机制冲突 | 已被替代 | 退役候选 |
| 维护成本 > 收益 | 低 ROI | 退役候选 |
| 引用的工具/模型已下线 | 失效 | 立即退役 |

### 8.3 退役流程（谨慎，需人工确认）

1. 标记 `[DEPRECATED]` + 原因 + 替代方案（如有）
2. 14 天宽限期内观察是否仍有依赖
3. **人工确认**后删除（Agent 可提议，但禁止自行删除 SOP/记忆）
4. 删除后更新 L1 索引，记录到变更日志

> **原则**：宁可保留并标注过时，也不误删有价值经验。删除是不可逆的，必须谨慎。

## 9. Meta-rule：规则自身的持续审核与进化

> 规则（包括本 SOP）也要迭代，防止规则腐化。源自持续改进（Kaizen）+ 规则版本管理。

### 9.1 规则版本管理
- 每个规则/SOP 必须有：版本号、创建时间、最后验证日期、状态、替代方案。
- 修改时 bump 版本号 + 在变更记录写明改了什么、为什么改。

### 9.2 定期复盘周期
- **每 30 天**：`memory_lint.py` 扫描结构一致性 + 低 ROI 条目复查。
- **每 90 天**：规则有效性复盘——规则是否还在指导正确行为？是否需进化？
- **触发式复盘**：规则导致失败/误判时，立即复盘修正。

### 9.3 防腐化机制
- 规则数量应有上限意识：过多规则本身就是负担，定期精简合并。
- 退役规则走 §8.3 流程（标记→宽限期→人工确认删除）。
- 每条规则应可追溯到**真实事件/失败/需求**，无来源的规则优先清理。

## 10. 典型坑

1. **官方也改核心框架**——不只 TUI 升级，`agent_loop`/`llmcore` 也可能动，merge 前必看 diff。
2. **工作区未提交导致 merge 失败/覆盖**——先 commit/stash。
3. **冲突期间文件带标记**——其他会话 code_run 会读到语法错误，尽快解决或 abort。
4. **不要依赖 `ga update`**——官方明确说用 git 手动控制。
5. **pyproject `version=0.1.0` 是占位值**——不能判断是否需升级，以 git commit 为准。
6. **多 TUI 并存**——tuiapp.py(v1)/tuiapp_v2.py(v2)/tui_v3.py(v3)，升级后明确启动哪个。
7. **备份只增不减会膨胀**——必须执行 §4.3 清理门禁，否则备份堆积。
8. **规则只加不减会腐化**——必须执行 §8 退役 + §9 自审，否则规则失效。

## 11. 已知限制

- 不实现全自动升级（核心框架冲突必须人工确认）。
- Git HTTPS fetch 可能被代理 reset，回退 SSH 或 GitHub API。
- 升级只保代码，不保运行中会话状态；其他会话需重启加载新代码。
- 退役/清理依赖人工最终确认，Agent 仅提议。
