## [Network / Proxy]
- 当前 Windows 环境启用了系统代理；pip 安装时必须显式绕过代理，否则会出现 `ProxyError` / `SSLEOFError`。
- 经验证可用的命令模板：`pip install --proxy= -i https://pypi.tuna.tsinghua.edu.cn/simple --trusted-host pypi.tuna.tsinghua.edu.cn <package>`。
- 用 Python `requests` 访问 GitHub API/raw 时也会因系统代理触发 `SSLEOFError`；已验证方案：设置 `os.environ['HTTPS_PROXY']=''` 并在 `requests.get(..., proxies={'http': None, 'https': None})` 中显式禁用。
- 学术调研 API 限流：Semantic Scholar（`api.semanticscholar.org`）免费层高频访问易触发 HTTP 429（需间隔重试或换源）；arXiv API 查询禁用 `OR`/`+`（导致空结果），用简洁关键词。

- 当前后端：主模型 `zhipu-glm`（国内 Coding Plan，Anthropic 协议）+ 备用模型 `kimi`（端点 `https://api.kimi.com/coding/v1`），固定顺序故障转移。

- GLM Coding 最佳实践（来源：docs.bigmodel.cn）：
  - 任务拆分：先完成小任务，再逐步扩展；复杂任务先进入 Plan Mode 规划。
  - 上下文管理：通过 `project.md` / `CLAUDE.md` 沉淀项目架构、目录约定、构建测试命令、命名规范；规则要具体、可执行。

  - 子代理与技能：跨模块/跨领域任务拆分子代理；将可复用流程封装为 Skill（如代码审查、文档生成）。
  - 外部扩展：通过 MCP 连接外部工具/数据，通过 Skill 定义使用方式与业务逻辑。
  - 自动化：对重复流程（如 PR 生成、文档同步、测试补充）建立自动化脚本或工作流。
  - Prompt 工程：明确角色、目标、输入/输出格式、边界条件；对复杂提示做迭代优化。
  - 权限与验证：敏感操作前确认权限；安装/修改 LLM 配置后做端到端调用验证。
